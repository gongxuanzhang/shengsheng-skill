# CLAUDE.md 更新规范

本文件定义 `CLAUDE.md` 更新时的内容标准和章节定位规则。

---

## 章节结构（当前项目）

| 章节 | 标题 | 更新场景 |
|------|------|---------|
| 一 | 项目概述 | 技术栈变更、项目定位变更 |
| 二 | 核心架构决策 | 新增/修改架构方案（截图方案、识别方案、通信方案等） |
| 三 | 状态机概要 | 状态枚举变更、转换规则变更 |
| 四 | API 设计 | 新增/修改 API 端点、请求/响应格式 |
| 五 | 闲管家开放平台集成 | 闲管家接口变更、新增 Webhook |
| 六 | 项目结构 | 新增/删除/移动文件、新增模块 |
| 七 | 买家 H5 页面 | 页面逻辑变更、轮询策略变更 |
| 八 | 管理后台 | 后台页面功能变更 |
| 九 | 待完成任务 | ⚠️ 不再使用，任务统一记录在 todo.md |
| 十 | 开发规范 | 新增代码约定、工具函数统一 |
| 十一 | 快速启动指南 | 启动参数/步骤变更 |
| 十二 | 已知问题 | 指向 docs/known-issues.md |
| 十三 | 设计说明 | 新增设计决策说明（兜底章节） |
| 十四 | 故障排查 | 新增排查经验 |

## 更新原则

### 1. 最小化变更

- 只改动与当前讨论直接相关的章节
- 不做全文重排、不改无关章节的措辞
- 保持现有的 Markdown 格式风格（表格对齐、代码块语言标注、缩进层级）

### 2. 新增内容的定位规则

| 变更类型 | 放在哪里 |
|---------|---------|
| 新增 API 端点 | 四、API 设计 → 对应子节 |
| 新增 DTO/Entity | 六、项目结构 → 对应目录 |
| 新增配置项 | 对应功能的章节（如闲管家配置放第五节） |
| 新增状态/枚举 | 三、状态机概要 |
| 新增文件 | 六、项目结构 → 目录树 |
| 新增设计决策 | 十三、设计说明（新增子标题） |
| 新增代码规范 | 十、开发规范 → "代码质量约定" |
| 新增故障排查 | 十四、故障排查 → 对应表格 |

### 3. 设计说明的格式

新增设计说明使用三级标题，格式：

```markdown
### {决策名称}

**问题**：描述需要解决的问题

**方案**：描述选择的方案

{代码示例（如有）}

**覆盖场景**：
- 场景 1
- 场景 2

**不影响的场景**：场景说明（如有）
```

### 4. 代码规范的格式

新增代码规范使用加粗前缀 + 说明：

```markdown
- **统一使用 `XxxUtil.method()`**，不要在各处重复实现 xxx 逻辑
```

### 5. 项目结构的格式

新增文件使用 `│   ├──` 树形格式 + 注释：

```markdown
│   ├── NewFile.java               # 一句话说明文件职责
```

---

## 不应更新 CLAUDE.md 的场景

- 纯粹的 bug 修复（不涉及设计变更）
- 变量重命名、import 整理
- 测试代码的修改
- 注释的修改
- 不影响架构的局部重构

## 关联文档发现与级联更新

### 1. 引用识别规则

更新 CLAUDE.md 时，需扫描以下形式的文档引用：

| 引用形式 | 示例 | 说明 |
|---------|------|------|
| Markdown 链接 | `[已知问题](docs/known-issues.md)` | 标准链接引用 |
| 指向性描述 | `指向 docs/known-issues.md` | 文本中直接提及路径 |
| 详见引用 | `详见 docs/api-spec.md` | 中文引用 |
| 参考引用 | `参考 docs/design.md 第三节` | 带章节指向 |
| 代码块路径 | `` `docs/xxx.md` `` | 行内代码引用 |
| 相对路径 | `./docs/xxx.md` 或 `../docs/xxx.md` | 相对路径引用 |

### 2. 引用过滤规则

- ✅ **处理**：项目内的相对路径，指向 `.md`、`.yaml`、`.yml`、`.json` 等文本文件
- ❌ **忽略**：外部 URL（`http://`、`https://`）
- ❌ **忽略**：不存在的文件路径
- ❌ **忽略**：二进制文件（`.png`、`.jar`、`.class` 等）
- ❌ **忽略**：源代码文件（`.java`、`.py`、`.ts` 等）— 这些由代码编辑流程维护

### 3. 关联性判断

扫描到引用后，需判断被引用文档是否与本次变更相关：

| 判断维度 | 相关（需更新） | 不相关（跳过） |
|---------|--------------|--------------|
| 模块匹配 | 本次变更了 API 设计，关联文档是 `docs/api-spec.md` | 本次变更了 API 设计，关联文档是 `docs/deployment.md` |
| 内容覆盖 | 关联文档中有本次变更涉及的章节 | 关联文档内容与本次变更完全无关 |
| 新增引用 | CLAUDE.md 中新增了对某文档的引用 | CLAUDE.md 中已有的引用，但内容未变 |

### 4. 关联文档更新原则

- **保持原文档风格**：每个文档可能有不同的格式，不要强制统一
- **最小化变更**：只修改与本次变更直接相关的部分
- **新增内容追加**：在关联文档合适位置追加，不打乱原有结构
- **删除对应内容**：如果 CLAUDE.md 删除了某个方案，关联文档中的对应描述也应移除或标记过时
- **版本一致性**：确保 CLAUDE.md 摘要描述与关联文档详细描述不矛盾

### 5. 常见关联文档类型及更新策略

| 关联文档类型 | 典型路径 | 更新策略 |
|------------|---------|---------|
| 已知问题文档 | `docs/known-issues.md` | 新增/解决已知问题时同步更新 |
| API 详细规格 | `docs/api-spec.md`、`docs/api/` | 新增/修改 API 端点时同步更新 |
| 设计文档 | `docs/design.md`、`docs/architecture/` | 架构决策变更时同步更新 |
| 部署文档 | `docs/deployment.md` | 配置项、环境要求变更时同步更新 |
| 变更日志 | `CHANGELOG.md` | 重大变更时追加记录 |
| 贡献指南 | `CONTRIBUTING.md` | 开发规范变更时同步更新 |
