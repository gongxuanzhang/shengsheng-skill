# todo.md 格式规范

本文件定义 `todo.md` 的内容标准，供 Skill 执行时参照。

---

## 文件整体结构

```markdown
# TODO — {项目名} 待办事项

> 最后更新: {YYYY-MM-DD}
> 配合 CLAUDE.md 阅读，本文件聚焦"要做什么"，CLAUDE.md 聚焦"架构是什么"。

---

## 🔴 上线必需

### 1. 标题
- **文件**: `路径`
- **现象/现状**: ...
- ...

---

## 🟡 应该修复的代码问题

### N. 标题
- ...

---

## 🟢 改进优化（可选分区，有需要时创建）

### N. 标题
- ...

---

## 📝 备忘

- 固定提醒事项（端口号、默认密码、设计意图等）
```

---

## 条目格式

每个 TODO 条目是一个三级标题（`###`），编号全局连续（跨分区递增）。

### 必填字段

| 字段 | 格式 | 说明 |
|------|------|------|
| 标题 | `### {N}. {动词短语}` | 简明描述要做什么 |
| 文件 | `- **文件**: \`path\`` | 涉及的主要文件，多个用 `+` 连接 |

### 可选字段（按需）

| 字段 | 适用场景 |
|------|---------|
| `- **现象/现状**:` | 描述当前问题或现状 |
| `- **影响**:` | 列出不修复的影响 |
| `- **建议**:` | 建议的解决方案 |
| `- **操作**:` | 具体执行步骤 |
| `- **重点验证**:` | 需要验证的关键点 |
| `- **注意**:` | 额外注意事项 |
| `- **目标**:` | 期望达到的状态 |
| `- **测试要求**:` | 需要的测试覆盖 |
| `- **落地原则**:` | 执行原则列表 |

### 示例：上线必需

```markdown
### 1. 生产环境配置
- **文件**: `src/main/resources/application.yml` + `client-python/config.yaml`
- **待配置项**:
  - `application.yml` → `app.base-url`: 改为公网域名（当前 `localhost:55588`）
  - `config.yaml` → `server.base_url` / `server.ws_url`: 改为实际服务器地址
- **注意**: H2 数据库生产环境建议换 MySQL
```

### 示例：代码问题

```markdown
### 4. 测试配置与主配置漂移
- **文件**: `src/test/resources/application-test.yml`
- **现象**: 测试配置仍有旧 key，当前测试通过依赖主配置兜底
- **影响**:
  - 测试环境不自洽
  - 后续配置项重构时可能出现假阳性
- **建议**:
  - 按当前 `AppProperties` 结构清理并重写测试配置
  - 让测试启动只依赖 `application-test.yml`
```

## 编号规则

1. 编号全局连续（1, 2, 3, ...），跨分区递增
2. 删除条目后，后续条目编号向前顺延
3. 新增条目编号 = 当前最大编号 + 1，追加到对应分区末尾
4. 备忘区不编号

---

## 分区分类标准

| 分区 | 标准 | 典型场景 |
|------|------|---------|
| 🔴 上线必需 | 不做就无法上线或上线后必出问题 | 生产环境配置、关键功能缺失、安全漏洞 |
| 🟡 应该修复 | 可以上线但会埋隐患 | 技术债务、非阻塞 bug、性能问题、代码质量 |
| 🟢 改进优化 | 锦上添花，可延后 | 体验优化、重构、新功能预研 |
| 📝 备忘 | 常驻提醒，不是任务 | 端口号、默认密码、设计意图 |

---

## 删除规则

- 删除时移除条目的完整内容（`### N. 标题` 到下一个 `###` 之前的所有行）
- 删除后重新编号，保持连续
- 如果某分区删空了，保留分区标题（不删分区）
- 更新文件头的"最后更新"日期
