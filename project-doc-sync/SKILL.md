# Project Doc Sync — 项目文档自动同步

在开发对话过程中，自动维护项目的 `todo.md`（待办清单）和 `CLAUDE.md`（行为指令集），确保项目文档与实际开发进度始终一致。

> **CLAUDE.md 定位**：行为指令集（类似给新工程师的 onboarding checklist），而非架构知识库。≤ 200 行。

---

## 职责

1. **todo.md 追加**：对话中讨论出新的待办事项 → 自动追加到 `todo.md`
2. **todo.md 删除**：某个 TODO 被确认完成 → 从 `todo.md` 中删除该条目，重新编号
3. **CLAUDE.md 更新**：对话中确认了新的开发规范、架构决策、踩坑经验 → 更新 `CLAUDE.md` 对应章节
4. **内容外置建议**：当 CLAUDE.md 内容过于详细或超过 200 行时，建议用户将详情移到 `docs/` 并用 `@` 引用

---

## 触发场景

### 触发 todo.md 追加

对话中出现以下信号时触发：

- 明确讨论出需要执行的任务清单（"我们需要做 X、Y、Z"）
- 用户要求 "记一下" / "加到 todo" / "这个后面做"
- 代码审查或排查中发现需要修复的问题
- 实现过程中发现衍生任务（"这个先不改，记下来"）

### 触发 todo.md 删除

- 某项任务的代码已实现并验证通过
- 用户明确确认 "这个做完了" / "已经解决"
- 代码审查中发现 TODO 对应的功能已经存在

### 触发 CLAUDE.md 更新

- 讨论并确认了新的架构决策（且该决策影响日常开发行为）
- 新增了开发规范、代码约定
- 新增了故障排查经验（踩坑记录）
- 构建/测试/部署命令发生变更
- 发现 Claude 反复犯同样的错误（需要记录为 Gotcha）
- 项目有非常规约定需要告知 Claude

### 不触发 CLAUDE.md 更新

- 新增了 API 端点、DTO、Entity、Controller、Service 文件（Claude 可以从代码读取）
- 新增了配置项（Claude 可以从配置文件读取）
- 修改了状态机、业务流程（Claude 可以从代码读取）
- 纯粹的代码编写（无新决策、无新任务）
- 用户只是在提问、查阅信息
- 修改了代码但不涉及架构变更（bug 修复、重命名等）

## 执行步骤

### todo.md 追加流程

1. **读取** 当前 `todo.md` 获取现有结构和最大编号
2. **判断分类**：新任务属于哪个分区（🔴 上线必需 / 🟡 应该修复 / 🟢 改进优化）
3. **编写条目**：按标准格式编写新条目（见 reference/todo-format.md）
4. **追加**：在对应分区末尾追加新条目，编号递增
5. **告知用户**：简要说明 "已追加 TODO-N: xxx 到 todo.md"

### todo.md 删除流程

1. **读取** 当前 `todo.md`
2. **定位** 已完成的条目
3. **删除** 该条目的完整内容（标题 + 所有子项）
4. **重新编号**：后续条目编号向前顺延，保持连续
5. **告知用户**：简要说明 "已从 todo.md 删除 TODO-N: xxx"

### CLAUDE.md 更新流程

0. **内容筛选**（新增 — 最关键的一步）：
   - 自问："没有这行，Claude 读代码后还会犯错吗？" → 不会就**不加**
   - Claude 可以从代码/配置文件直接读取的信息 → **不加**
   - 语言/框架的默认约定 → **不加**
   - 只有**影响日常开发行为且无法从代码推断**的内容才写入
1. **读取** 当前 `CLAUDE.md` 完整内容
2. **定位** 需要更新的章节（按通用章节模型匹配，见 reference/claude-md-format.md）
3. **更新内容**：
   - 使用**指令风格**编写（"做 X 时必须 Y"，而非"X 是什么"）
   - 新增章节 → 追加到合适位置
   - 修改现有内容 → 原地替换
4. **长度检查**：
   - 统计更新后 CLAUDE.md 总行数
   - ≤ 150 行：正常
   - 151-200 行：提醒用户"接近上限，建议审视是否有可外置的内容"
   - \> 200 行：**必须精简**——将详细内容外置到 `docs/` 并用 `@` 引用
5. **渐进式披露**：
   - 如果待写入内容超过 5 行，考虑外置到 `docs/` 或 `.claude/rules/`
   - CLAUDE.md 中只保留一行指令摘要 + `@` 引用
6. **扫描关联文档**：检查是否需要同步更新被引用的外部文档
7. **告知用户**：简要说明 "已更新 CLAUDE.md: xxx"

### 关联文档更新流程

> **方向原则**：详细内容应该在外部文档中，CLAUDE.md 只引用它们。

当 CLAUDE.md 更新触发后，执行以下步骤：

1. **扫描引用**：读取 `CLAUDE.md`，提取所有文档引用（`@docs/xxx.md`、`[文字](路径)`、`详见 docs/xxx.md`）
2. **过滤范围**：只处理项目内的相对路径文档（排除外部 URL），只处理实际存在的文件
3. **判断关联性**：逐一判断被引用文档是否与本次变更相关
4. **读取并更新**：对每个相关的关联文档，按照该文档自身的格式风格进行最小化更新
5. **内容外置建议**：如果发现 CLAUDE.md 中有内容过于详细（API 规格、DTO 列表等），建议用户将其移到外部文档并用 `@` 引用
6. **告知用户**：简要说明 "已同步更新关联文档: docs/xxx.md"

## 输出标准

### todo.md 条目格式

```markdown
### {编号}. {标题}
- **文件**: `相关文件路径`
- **现象/现状**: 一句话描述当前问题或现状
- **影响**: 列出影响点
- **建议**: 列出建议的解决方案
```

### todo.md 分区规则

| 分区 | 标记 | 适用范围 |
|------|------|---------|
| 上线必需 | 🔴 | 不做就不能上线的任务 |
| 应该修复的代码问题 | 🟡 | 技术债务、非阻塞性问题 |
| 改进优化 | 🟢 | 锦上添花、可延后 |

### CLAUDE.md 更新规则

#### 内容分类规则

| 内容类型 | 是否写入 CLAUDE.md | 说明 |
|---------|-------------------|------|
| 构建/测试/部署命令 | ✅ 写入 | Claude 无法猜到 |
| 非常规代码约定 | ✅ 写入 | 与语言/框架默认不同的规则 |
| 影响开发行为的架构决策 | ✅ 写入 | 代码中不明显的 why |
| 踩坑记录 / Gotchas | ✅ 写入 | Claude 容易重复犯错的行为 |
| 团队特有工作流 | ✅ 写入 | 分支策略、PR 模板、发布流程 |
| API 详细规格 | ❌ 不写入 | Claude 可以读代码 |
| DTO/Entity 字段列表 | ❌ 不写入 | Claude 可以读代码 |
| 状态机完整定义 | ❌ 不写入 | Claude 可以读代码 |
| 配置 YAML 示例 | ❌ 不写入 | Claude 可以读配置文件 |
| 完整项目文件树 | ❌ 不写入 | Claude 可以用 Glob 查看 |

#### 指令风格要求

- 每条内容都应是**可执行的行为指令**
- 正确："新增 API 端点时，必须同时添加集成测试"
- 错误："项目使用 Spring Boot 3.2 框架"（知识描述，Claude 读 pom.xml 就知道）

#### 渐进式披露要求

- 超过 5 行的详细内容 → 外置到 `docs/` 或 `.claude/rules/`
- CLAUDE.md 只保留一行指令摘要 + `@docs/xxx.md` 引用

### 关联文档更新规则

- **详情在外部，引用在 CLAUDE.md**：CLAUDE.md 保持精简，详细内容放在被引用的外部文档中
- **只更新已存在的文件**：不自动创建新的关联文档
- **尊重原文档格式**：每个关联文档可能有自己的格式风格，更新时保持其原有风格
- **变更范围最小化**：只更新关联文档中与本次变更直接相关的部分
- **更新后汇报**：告知用户所有被更新的文档列表和更新内容摘要

### 通用规则

- **只改动该改的**：不做全文重写，最小化变更
- **保持编号连续**：删除条目后重新编号
- **中文优先**：与项目现有文档风格一致
- **不主动创建文件**：只更新已存在的 `todo.md`、`CLAUDE.md` 和关联文档，不自动创建新文件
- **更新后告知**：每次改动后用一句话告知用户改了什么
- **200 行上限**：CLAUDE.md 总行数不超过 200 行，超过时必须精简或外置
