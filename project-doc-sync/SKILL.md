# Project Doc Sync — 项目文档自动同步

在开发对话过程中，自动维护项目的 `todo.md`（待办清单）、`CLAUDE.md`（架构方案）以及 CLAUDE.md 引用的关联文档，确保所有项目文档与实际开发进度始终一致。

---

## 职责

1. **todo.md 追加**：对话中讨论出新的待办事项 → 自动追加到 `todo.md`
2. **todo.md 删除**：某个 TODO 被确认完成 → 从 `todo.md` 中删除该条目，重新编号
3. **CLAUDE.md 更新**：对话中确认了新的设计方案、架构决策 → 更新 `CLAUDE.md` 对应章节
4. **关联文档级联更新**：更新 `CLAUDE.md` 时，扫描其中引用的关联文档，同步更新受影响的文档

---

## 触发场景

### 触发 todo.md 追加

对话中出现以下信号时触发：

- 明确讨论出需要执行的任务清单（"我们需要做 X、Y、Z"）
- 用户要求 "记一下" / "加到 todo" / "这个后面做"
- 代码审查或排查中发现需要修复的问题
- 实现过程中发现衍生任务（"这个先不改，记下来"）

### 触发 todo.md 删除

- 某项任务的代码已实现并验证通过
- 用户明确确认 "这个做完了" / "已经解决"
- 代码审查中发现 TODO 对应的功能已经存在

### 触发 CLAUDE.md 更新

- 对话中确认了新的架构决策（如新增模块、修改通信协议）
- 新增了 API 端点、DTO、配置项
- 修改了状态机、业务流程
- 新增了开发规范、代码约定
- 新增了故障排查经验
- 修改了项目结构（新增/删除文件）

### 触发关联文档级联更新

当 CLAUDE.md 的更新涉及以下情况时，必须同时更新被引用的关联文档：

- `CLAUDE.md` 中通过链接引用了其他文档（如 `详见 [xxx](docs/xxx.md)`、`指向 docs/xxx.md`）
- 被引用文档中的内容与本次设计变更直接相关
- 新增/修改的设计方案在关联文档中有对应的详细描述
- 修改了某个模块的架构，而该模块在关联文档中有独立的详细说明

### 不触发的场景

- 纯粹的代码编写（无新决策、无新任务）
- 用户只是在提问、查阅信息
- 修改了代码但不涉及架构变更（bug 修复、重命名等）

## 执行步骤

### todo.md 追加流程

1. **读取** 当前 `todo.md` 获取现有结构和最大编号
2. **判断分类**：新任务属于哪个分区（🔴 上线必需 / 🟡 应该修复 / 🟢 改进优化）
3. **编写条目**：按标准格式编写新条目（见 reference/todo-format.md）
4. **追加**：在对应分区末尾追加新条目，编号递增
5. **告知用户**：简要说明 "已追加 TODO-N: xxx 到 todo.md"

### todo.md 删除流程

1. **读取** 当前 `todo.md`
2. **定位** 已完成的条目
3. **删除** 该条目的完整内容（标题 + 所有子项）
4. **重新编号**：后续条目编号向前顺延，保持连续
5. **告知用户**：简要说明 "已从 todo.md 删除 TODO-N: xxx"

### CLAUDE.md 更新流程

1. **读取** 当前 `CLAUDE.md` 完整内容
2. **定位** 需要更新的章节（按一级/二级标题匹配）
3. **更新内容**：
   - 新增章节 → 追加到合适位置
   - 修改现有内容 → 原地替换
   - 新增字段/配置 → 追加到对应表格
4. **保持风格一致**：使用项目现有的 Markdown 风格（表格、代码块、缩进）
5. **扫描关联文档**：检查本次变更是否涉及 CLAUDE.md 中引用的其他文档（触发关联文档级联更新）
6. **告知用户**：简要说明 "已更新 CLAUDE.md 第 X 节: xxx"

### 关联文档级联更新流程

当 CLAUDE.md 更新触发后，执行以下步骤：

1. **扫描引用**：读取 `CLAUDE.md`，提取所有文档引用，包括：
   - Markdown 链接：`[文字](路径)` 格式
   - 直接路径引用：`详见 docs/xxx.md`、`指向 docs/xxx.md`、`参考 docs/xxx.md`
   - 代码块内的路径引用：`` `docs/xxx.md` ``
2. **过滤范围**：只处理项目内的相对路径文档（排除外部 URL），只处理实际存在的文件
3. **判断关联性**：逐一判断被引用文档是否与本次变更相关：
   - **相关**：被引用文档描述的模块/功能与本次变更直接相关
   - **不相关**：被引用文档描述的内容与本次变更无关
4. **读取并更新**：对每个相关的关联文档：
   - 读取文档完整内容
   - 定位需要更新的章节
   - 按照该文档自身的格式风格进行更新
   - 保持最小化变更原则
5. **一致性校验**：确保 CLAUDE.md 中的引用描述与关联文档的实际内容一致（如 CLAUDE.md 说"详见 docs/api.md 的认证章节"，则 api.md 中必须有对应内容）
6. **告知用户**：简要说明 "已同步更新关联文档: docs/xxx.md（更新了 xxx 部分）"

## 输出标准

### todo.md 条目格式

```markdown
### {编号}. {标题}
- **文件**: `相关文件路径`
- **现象/现状**: 一句话描述当前问题或现状
- **影响**: 列出影响点
- **建议**: 列出建议的解决方案
```

### todo.md 分区规则

| 分区 | 标记 | 适用范围 |
|------|------|---------|
| 上线必需 | 🔴 | 不做就不能上线的任务 |
| 应该修复的代码问题 | 🟡 | 技术债务、非阻塞性问题 |
| 改进优化 | 🟢 | 锦上添花、可延后 |

### CLAUDE.md 更新规则

- 只更新与讨论内容直接相关的章节，不改动无关部分
- 新增 API → 更新"四、API 设计"
- 新增实体/DTO → 更新"六、项目结构"
- 新增配置项 → 更新对应集成章节
- 新增设计决策 → 更新"十三、设计说明" 或 "二、核心架构决策"
- 新增开发规范 → 更新"十、开发规范"
- 新增故障排查 → 更新"十四、故障排查"

### 关联文档更新规则

- **先 CLAUDE.md，后关联文档**：总是先更新 CLAUDE.md，再级联更新关联文档
- **引用发现**：扫描 CLAUDE.md 中所有形如 `[文字](相对路径)`、`` `相对路径` ``、`详见/参考/指向 相对路径` 的引用
- **只更新已存在的文件**：不自动创建新的关联文档，只更新已存在的文件
- **尊重原文档格式**：每个关联文档可能有自己的格式风格，更新时保持其原有风格
- **内容一致性**：确保 CLAUDE.md 中对关联文档的描述与关联文档实际内容一致
- **变更范围最小化**：只更新关联文档中与本次变更直接相关的部分
- **更新后汇报**：告知用户所有被级联更新的文档列表和更新内容摘要

### 通用规则

- **只改动该改的**：不做全文重写，最小化变更
- **保持编号连续**：删除条目后重新编号
- **中文优先**：与项目现有文档风格一致
- **不主动创建文件**：只更新已存在的 `todo.md`、`CLAUDE.md` 和关联文档，不自动创建新文件
- **更新后告知**：每次改动后用一句话告知用户改了什么
- **级联更新告知**：如果触发了关联文档更新，需额外列出所有被更新的文档路径
